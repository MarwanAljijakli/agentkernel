## Context and problem statement

Describe the current state, the concrete problem, and why a decision is needed now. Distinguish observed facts from assumptions and future goals. Link relevant issues, threat-model sections, and evidence.

## Security impact

Identify affected assets, adversaries, trust boundaries, invariants, new attack surface, fail-open/fail-closed behavior, and required negative tests. State whether the [threat model](../security/threat-model.md) must change.

## Data and privacy impact

Describe data classes, provenance, retention, external transmission, redaction, deletion/lineage, tenant boundaries, and any new third-party processing.

## Compatibility and migration

Describe schema/API/protocol compatibility, version negotiation, upgrade path, downgrade limitations, data migration, and behavior for older readers/writers.

## Operational impact

Describe deployment, configuration, observability, resource, failure-recovery, rollback, and incident-response effects.

## Research and reproducibility impact

Describe changes to benchmark comparability, datasets, recorded inputs, split integrity, metrics, thresholds, seeds, or replay fidelity.

## Validation evidence

List the automated tests, property checks, clean-environment flow, benchmarks, security review, and retained artifacts required before the decision can be called implemented. An accepted ADR is not itself implementation evidence.

## Rollout and rollback

State how the change is introduced, feature/version gates, how failure is detected, and the safe path to revert or supersede it. Never claim rollback for an irreversible effect.

## Revisit triggers

List evidence or events that require reconsideration: performance thresholds, incidents, new backends, standards changes, incompatibility, or invalidated assumptions.
